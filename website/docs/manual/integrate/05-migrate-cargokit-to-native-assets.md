# Migrate from Cargokit to Native Assets

This page describes how to move an existing flutter_rust_bridge project from the default Cargokit integration backend to the Native Assets backend.

Native Assets only changes how the Rust library is built and bundled. It does not change the flutter_rust_bridge core code generator, Rust API shape, or Dart API shape.

## When to migrate

Consider migrating when all of these are true:

- Your project can require a Flutter/Dart SDK that supports build hooks and code assets.
- You want Dart/Flutter build hooks to build and bundle the Rust library.
- You can test every platform that your application or package supports after the migration.

Keep Cargokit when you need older Flutter/Dart SDKs, have platform-specific Cargokit customization that is not yet easy to express in a build hook, or need the lowest-risk path for an existing production project.

## Before you start

Create a git branch and make sure the existing Cargokit project builds before changing the backend.

```bash
git switch -c migrate-to-native-assets
flutter_rust_bridge_codegen generate
flutter run
```

If the project has platform-specific edits under `rust_builder/`, `cargokit/`, `android/`, `ios/`, `linux/`, `macos/`, or `windows/`, review them before deleting anything. The Native Assets backend does not use the Cargokit build tool, Gradle plugin, CocoaPods scripts, or CMake plugin registration.

## Run the Native Assets integration

From the Flutter package root, run:

```bash
flutter_rust_bridge_codegen integrate --integration-backend native-assets
```

This adds the Native Assets backend files and dependencies. In particular, the generated `hook/build.dart` uses `flutter_rust_bridge_hooks`, which wraps `native_toolchain_rust` to build the Rust crate.

For a standard app, the generated hook should look similar to:

```dart
import 'package:flutter_rust_bridge_hooks/flutter_rust_bridge_hooks.dart';

void main(List<String> args) async {
  await build(args, (input, output) async {
    await const FlutterRustBridgeNativeAssetsBuilder(
      cratePath: 'rust',
    ).run(input: input, output: output);
  });
}
```

## Check the Rust crate

`native_toolchain_rust` needs the Rust crate to produce both dynamic and static library artifacts. In `rust/Cargo.toml`, keep:

```toml
[lib]
crate-type = ["cdylib", "staticlib"]
```

Also keep a pinned Rust toolchain in `rust/rust-toolchain.toml`. Use a concrete version or date, not a floating channel such as `stable`, `beta`, or `nightly`.

```toml
[toolchain]
channel = "1.93.1"
targets = [
  "aarch64-apple-darwin",
  "aarch64-apple-ios",
  "aarch64-apple-ios-sim",
  "aarch64-linux-android",
  "aarch64-pc-windows-msvc",
  "aarch64-unknown-linux-gnu",
  "armv7-linux-androideabi",
  "x86_64-apple-darwin",
  "x86_64-apple-ios",
  "x86_64-linux-android",
  "x86_64-pc-windows-msvc",
  "x86_64-unknown-linux-gnu",
]
```

Adjust the target list to the platforms your project actually supports.

## Remove the old Cargokit backend

After the Native Assets hook is present, remove the old Cargokit build backend from the Dart package.

For an app generated by flutter_rust_bridge, this usually means:

- Remove the generated `rust_builder/` package.
- Remove the `rust_builder` path dependency from `pubspec.yaml`.
- Remove `rust_builder/cargokit/**` from `analysis_options.yaml` if it is only there to silence Cargokit analyzer output.

For a Flutter package generated by flutter_rust_bridge, this usually means:

- Remove the generated `cargokit/` directory.
- Remove Flutter plugin platform entries that only existed for the Cargokit `ffiPlugin` scaffold.
- Remove Linux or Windows generated plugin registration entries that reference the old FFI plugin, if they were checked in by the package example.

Do not delete your Rust crate, `lib/src/rust/frb_generated*.dart`, or your own Dart/Rust source files. Those are shared by both backends.

## Update dependencies

Run:

```bash
flutter pub get
```

Then check `pubspec.yaml`. A migrated Native Assets project should depend on `flutter_rust_bridge` and `flutter_rust_bridge_hooks`. It should not still depend on the old generated `rust_builder` package unless you intentionally keep Cargokit side by side during a short transition.

For published packages, ensure hook dependencies are normal `dependencies`, not `dev_dependencies`, because build hooks run for packages that depend on your package.

## Verify each platform

Run the bridge generator first:

```bash
flutter_rust_bridge_codegen generate
```

Then test the target platforms you support. For example:

```bash
flutter run -d macos
flutter build apk
flutter build ios --no-codesign
flutter build linux
flutter build windows
```

Use the platform commands that match your project. The first build may take longer because the hook builds the Rust crate through Cargo.

## iOS and macOS notes

The Native Assets backend does not generate a Swift Package. Instead, it avoids the old Flutter `ffiPlugin` scaffold and lets Dart/Flutter build hooks provide the compiled Rust code asset.

This can avoid Swift Package Manager warnings that apply to Flutter plugin packages, but it is not the same thing as adding `Package.swift` support to the Cargokit backend.

## Troubleshooting

If Flutter cannot find the native library at runtime, check that:

- `hook/build.dart` exists in the Dart package root.
- `flutter_rust_bridge_hooks` is in `dependencies`.
- `rust/Cargo.toml` has both `cdylib` and `staticlib` crate types.
- `rust/rust-toolchain.toml` pins the Rust toolchain and includes the target you are building.
- The old Cargokit dependency has been removed from `pubspec.yaml`.

If Cargo fails during the hook build, run the same Flutter command with verbose logging:

```bash
flutter run -v
```

The hook invokes Cargo through `native_toolchain_rust`, so ordinary Rust target, linker, and platform SDK errors still apply.
